# Scraper 模块设计方案

> 为项目增加自动爬取能力，从互联网发现漫威角色/团队等数据，转化为现有管道可消费的 batch 格式。
>
> **本文档已反映 v2 数据层设计：爬虫输出 JSON → 管道消费 JSON，不修改 Python 源文件。**

---

## 目标

- 自动从 Wikipedia / Marvel Fandom Wiki 发现新角色
- 抓取结构化信息（真实姓名、物种、首次出场、能力等）
- 推断人物关系（团队归属、敌对/盟友）
- 输出为 JSON 文件，与现有 `run.py` 管道衔接
- 与现有管道衔接：爬完 → 通过 `--scraped` 导入

---

## 架构概览

```
用户 → scripts/scraper/discover.py --theme street-heroes [多个 theme]
  或 → scripts/scraper/discover.py --random 2
                      ↓
         对每个 theme 分别搜索 + 抓取
                      ↓
         合并结果 → 去重 → 推断关系
                      ↓
         scheduled_data/crawled/YYYY-MM-DD_run-xxx.json    ← 中间产物
                      ↓
         python3 scripts/collector/run.py --scraped scheduled_data/crawled/xxx.json
                      ↓
         collector_helper.generate_batch_from_data()
                      ↓
         Cypher 文件 → scheduled_data/batch_NNN/ → 双库同步
```

## 数据层设计

```
scheduled_data/
├── curated/                              ← 手工审核的批次（JSON)
│   ├── 001_spider-verse-and-more.json
│   └── 002_cosmic-expansion.json
├── crawled/                              ← 爬虫输出（不在 BATCHES 中自动加载）
│   ├── 2026-06-29_street-heroes.json     ← 单主题
│   ├── 2026-06-30_cosmic-mutant.villains.json  ← 多主题
│   └── 2026-07-01_random-2.json          ← 随机主题
├── batch_001/                            ← 已有的 Cypher 文件
├── batch_002/
└── index.json
```

**重要原则：**

| 层 | 内容 | 加载方式 |
|----|------|----------|
| `curated/` | 手工编写、审核通过的批次 | `data_definitions.py` 启动时自动加载为 `BATCHES` |
| `crawled/` | 爬虫自动产出，未经审核 | 仅通过 `run.py --scraped` 按需处理 |

`scripts/collector/data_definitions.py` 变为纯加载器，不含任何具体数据集：

```python
# data_definitions.py — 纯加载器
def _load_batches():
    if not BATCHES_DIR.exists():
        return []
    batches = []
    for f in sorted(BATCHES_DIR.glob("*.json")):
        batches.append(json.loads(f.read_text(encoding="utf-8")))
    return batches

BATCHES = _load_batches()
```

## 模块文件结构

```
scripts/scraper/
├── __init__.py
├── discover.py          # CLI 入口：根据主题发现角色
├── fetch_wiki.py        # Wikipedia 页面抓取 + infobox 解析
```

## 各模块职责

### `discover.py` — CLI 入口

**设计为 cron 友好**：支持单主题、多主题、随机主题三种模式。

```
# 单主题
python3 scripts/scraper/discover.py --theme street-heroes --count 15 --merge

# 多主题（逗号分隔）
python3 scripts/scraper/discover.py --theme street-heroes,cosmic,mfantasy --count 10 --merge

# 随机 2 个主题（cron 推荐用法）
python3 scripts/scraper/discover.py --random 2 --count 10 --merge
```

**参数说明：**

```
  --theme       主题，可指定多个（逗号分隔）如 street-heroes,cosmic
  --batch-name  生成的 batch 名称（多主题时自动生成）
  --count       每个主题发现多少个角色（默认 10）
  --list-themes 列出所有可用主题
  --random N    从所有主题中随机选 N 个爬取
  --merge       爬完后自动调用 run.py --scraped 导入
```

**多主题模式：**
- `--theme` 逗号分隔，按顺序逐个爬取
- 角色合并去重（同名角色只保留一份）
- 关系在合并后的全集上推断，跨主题角色可互相关联
- 输出文件：`scheduled_data/crawled/YYYY-MM-DD_theme1-theme2.json`

**随机模式：**
- `--random N`：从 `config.py` 定义的所有主题中随机选 N 个
- 适合 cron 定时任务（每次爬不同内容）
- 与 `--theme` 互斥

**预定义主题列表（`config.py`）：**

| 主题 | 搜索源 | 说明 |
|------|--------|------|
| `street-heroes` | Wikipedia "List of street-level superheroes" + Fandom | 夜魔侠、惩罚者、铁拳等 |
| `cosmic` | Wikipedia "Cosmic entities" + Fandom | Galactus、Celestials、Living Tribunal 等 |
| `mutants` | Wikipedia "List of Marvel mutants" | X-Men 遗漏成员、反派 |
| `villains` | Wikipedia "List of Marvel Comics villains" | 各反派 |
| `magic` | Wikipedia "Marvel magic users" | 奇异博士相关 |
| `mcu-only` | Wikipedia "Characters of the Marvel Cinematic Universe" | 仅 MCU 已出场角色 |
| `avengers-expand` | Wikipedia "List of Avengers members (comics)" | 更多复仇者 |
| `xmen-expand` | Wikipedia "List of X-Men members" | 更多 X-Men |

### `fetch_wiki.py` — Wikipedia 爬取

**功能：**
1. 根据主题构建 Wikipedia 搜索/分类页面 URL
2. 抓取页面，提取角色列表链接
3. 逐个抓取角色页面的 infobox
4. 从 infobox 解析结构化字段：

```
| 中文名 / 别名 → name
| 译名          → name (Chinese)
| 本名 / 真名   → real_name
| 种类 / 物种   → species
| 初次出现      → first_appearance
| 能力          → abilities
| 团队          → team_affiliations (用于关系推断)
| 主要敌人      → notable_enemies (用于关系推断)
```

5. 支持 Wikipedia 中文版（zh.wikipedia.org）获取中文名

**输出：** 人物 dict 列表，格式与 `data_definitions.py` 的 character 定义一致。

### `fetch_fandom.py` — Marvel Fandom Wiki 爬取

Fandom Wiki 比 Wikipedia 信息更丰富，infobox 字段更详细。作为 Wikipedia 的补充源。

**仅当 Wikipedia 抓取不到时使用**（降级策略）。

### `relationship.py` — 关系推断

根据抓取的数据自动推断关系：

```python
# 输入：角色列表（含 team_affiliations 字段）
# 输出：relationship tuples

推断规则：
1. 同一团队 → 成员关系 + 角色间盟友关系
2. 已知敌对 → 敌人关系（从 notable_enemies 提取）
3. 已知亲属 → 亲属关系（从 relatives 字段提取）
4. 已有角色之间的连接 → 自动复用
```

**去重：** 连接已存在的 `fixed/` + `scheduled_data/`，跳过已入库的角色。

### `output.py` — 输出

爬取结果写出为 `scheduled_data/crawled/` 目录下：

| 模式 | 文件命名 |
|------|----------|
| 单主题 | `YYYY-MM-DD_theme.json` |
| 多主题 | `YYYY-MM-DD_theme1-theme2.json` |
| 随机主题 | `YYYY-MM-DD_random-N.json` |

JSON 格式兼容 BATCHES 定义：

```json
{
  "name": "街头英雄扩展",
  "characters": [...],
  "teams": [...],
  "relationships": [
    ["Daredevil", "Character", "盟友", "Punisher", "Character", true]
  ]
}
```

**关系表示**：JSON 中用数组代替 Python tuple（下游 `generate_batch()` 通过索引 `rel[:5]`、`rel[5]` 访问，数组行为一致）。

## 与现有管道的衔接

```
爬虫 → scheduled_data/crawled/batch_XXX.json
          ↓
  run.py --scraped scheduled_data/crawled/batch_XXX.json
          ↓
  collector_helper.generate_batch_from_data()    ← 新增函数
          ↓
  generate_batch 核心逻辑（去重、ID 分配、Cypher 生成）
          ↓
  双库同步（cloud + local）
```

日常流程：

```bash
# 1. 爬取 15 个街头英雄 + 自动导入
python3 scripts/scraper/discover.py --theme street-heroes --count 15 --merge

# 2. 爬取多个主题 + 自动导入
python3 scripts/scraper/discover.py --theme street-heroes,cosmic,villains --count 10 --merge

# 3. 随机 3 个主题 + 自动导入（cron 推荐）
python3 scripts/scraper/discover.py --random 3 --count 10 --merge

# 4. 先生成 JSON，再手动处理
python3 scripts/scraper/discover.py --theme street-heroes --count 15
python3 scripts/collector/run.py --scraped scheduled_data/crawled/2026-06-29_street-heroes.json
```

`--merge` 在爬虫内部调用 `run.py --scraped <path>` 来处理新爬取的批次。

## Cron 自动化

推荐在 crontab 中设置周任务，每次随机抓取不同主题，避免主题偏好导致数据偏斜。

### 推荐配置

```cron
# 每周日凌晨 3 点，随机爬 2 个主题，各 10 个角色，自动导入
0 3 * * 0 cd /path/to/marvel-universe-data && \
  python3 scripts/scraper/discover.py --random 2 --count 10 --merge
```

### 为什么不固定主题

set：每次爬同一主题会：
1. 新角色越来越少（已有角色被去重跳过）
2. 数据库内容偏斜（某类角色坐满，其他类型空缺）
3. 失去"意外发现"的价值

而 `--random N` 每次随机选 N 个，长期来看各主题数据分布均匀，且每次都有新鲜感。

### 注意事项

- **频率建议**：每周 1-2 次（Wikipedia 和 Fandom 的内容更新不需要天级同步）
- **`--count` 不宜过大**：建议每主题 5-15 个，避免一次跑太久
- **`--merge` 必须加**：否则爬了不导入等于白爬
- **爬虫缓存**：`CACHE_DIR` 会让同一页面在缓存有效期内不重复请求，多次运行也不浪费
- **失败容忍**：某个主题的某个页面抓取失败自动跳过，不阻塞整次运行

## 去重策略

| 层级 | 方式 |
|------|------|
| 发现时 | 跳过已有 `name_en`（从 existing 索引查） |
| 输出时 | JSON 中标记 duplicate=False，已存在的设 skip |
| 处理时 | `collector_helper.generate_batch()` / `generate_batch_from_data()` 内部二次去重（幂等） |

### 现有角色列表如何获取

`collector_helper.get_all_existing_entities()` 已有这个能力：

```python
existing = get_all_existing_entities()
# existing["Character"] = {"Iron Man": "c1", "Spider-Man": "c100", ...}
```

scraper 可以直接复用这个函数。

## 反爬策略

在 `config.py` 中统一配置：

```python
REQUESTS_DELAY = 2.0       # 每次请求间隔（秒）
USER_AGENT = "..."         # 合理 UA
USE_PLAYWRIGHT = False     # 静态页面用 requests，动态用 playwright（降级）
CACHE_DIR = "scheduled_data/crawled_cache/"  # 缓存 HTML，避免重复请求
TIMEOUT = 15               # 请求超时（秒）
```

- 每页间隔 2+ 秒（可控）
- 缓存已抓取的 HTML，断点续爬
- 失败自动重试 1 次，失败则跳过（不阻塞）

## 增量可视

### 背景

v1 机制下，每次数据收集的增量变更只有 stdout 输出和 REPORT.md 中可覆写的「最新批次」段落，没有持久化的历史记录。v2 引入了 CHANGELOG.md 和 index.json 的 `scraped_runs` 字段来解决这个问题。

**文件位置**: 项目根目录 `CHANGELOG.md`

```markdown
### ✅ 新增实体（+5）

- c1008 — White Tiger — Angela Del Toro  |  人类
- c1009 — Cloak — Tyrone Johnson  |  变异人

**角色**: Daredevil（c3）, Punisher（c12）, Luke Cage（c15）
```

### index.json 的 `scraped_runs` 字段

**背景分析**: index.json 在新机制下**不可删除**。它在新架构中的角色是**状态存储**而非数据源。

| 字段 | 作用 | 留存 |
|------|------|------|
| `next_ids` | ID 分配器（cNNN 等） | ✅ 必须 |
| `import_log.cloud` / `.local` | 双库同步账本 | ✅ 必须 |
| `completed_batches` | 已生成 batch 号列表 | ✅ 保留 |
| `total_entities_added` | 累计计数 | ✅ 保留（离线参考） |
| `latest_batch` | 最新批次号 | ✅ 保留 |

新增 `scraped_runs` 数组提供从 crawled JSON → batch 编号的追溯关系：

```json
{
  "scraped_runs": [
    {
      "id": "2026-06-29_street-heroes",
      "source": "crawled/2026-06-29_street-heroes.json",
      "batch_num": 3,
      "time": "2026-06-29T10:30:00",
      "entities_added": 5,
      "entities_skipped": 2,
      "relationships_added": 8,
      "relationships_skipped": 1
    }
  ]
}
```

**谁读 index.json**:

| 文件 | 函数 | 读什么字段 |
|------|------|-----------|
| `collector_helper.py` | `get_batch_state()` | 全部 |
| `collector_helper.py` | `get_next_ids()` | `latest_batch`, `next_ids` |
| `collector_helper.py` | `generate_batch()` | `completed_batches` |
| `collector_helper.py` | `generate_report()` | `completed_batches`, `latest_batch` |
| `collector_helper.py` | `print_status()` | `completed_batches` |
| `run.py` | `cmd_status()` / `cmd_run()` | `completed_batches`, `import_log` |
| `sync_to_local.py` | `compute_pending()` | `completed_batches`, `import_log` |

### 使用方式

1. **日常增量查看**: `cat CHANGELOG.md` — 按时间倒序列出每次收集的新增内容
2. **快速统计**: `python3 scripts/collector/run.py --status` — 查看同步状态（不变）
3. **详细报告**: `REPORT.md` 的「最新批次」段落（不变）
